# 项目进展大白话说明

这份文档是给人看的，不是给机器看的。

它用大白话说明：这个项目现在到底在做什么、每一步为什么要做、运行结果代表什么、为什么有些数据进 clean_data，有些数据必须留在 review_queue。

如果你是刚接手这个项目的人，或者只是想快速看懂最近这一串 348N 任务，不用从一堆 RESULT / QA / task 文档里考古，先看这份。

---

## 1. 这个项目现在在做什么

当前主线不是“重新抽 PDF”，也不是“训练模型”。

当前主线是：

```text
把已有的 Excel 抽取结果读进来，判断哪些数据可信，哪些数据需要人工复核，最后形成可审计的 clean_data 和 review_queue。
```

也就是说，我们现在做的是一个金融文档抽取结果的审计系统。

它的核心流程是：

```text
intake -> audit -> review -> delivery
```

大白话解释：

```text
intake = 先把 Excel 读进来，看每一行是什么东西
audit = 检查单位、期间、证据、估值等有没有问题
review = 决定这行能不能进 clean_data，还是要进 review_queue
delivery = 以后正式交付，目前 gate 还关着
```

当前最重要的原则是：

```text
宁可让数据进 review_queue，也不要把不确定的数据错误放进 clean_data。
```

因为金融数据里，漏报可以补，错报会害人。项目不是为了让数字好看，而是为了让结果可解释、可审计、可回滚。

---

## 2. clean_data 和 review_queue 是什么

### clean_data

`clean_data` 是系统认为相对可信、可以作为后续结构化数据使用的行。

它必须很保守。

不是所有看起来像数据的东西都能进去。

### review_queue

`review_queue` 是需要人工复核的行。

这些行可能是：

```text
证据不够强
单位不明确
期间不明确
只是说明文字
是测试集说明
是字段字典
是图表索引
是校验行
是模型不确定的数据
```

review_queue 大并不一定坏。只要它可解释，就比把脏数据塞进 clean_data 好。

---

## 3. 为什么要分很多轮任务

这个项目不是一次性“把所有规则写完”。

原因很简单：真实 Excel 太乱了。

不同工具生成的 Excel 结构可能完全不一样：

```text
有的是普通宽表
有的是指标汇总表
有的是长表 testset
有的是字段字典
有的是 README
有的是图表索引
有的是校验表
```

如果一次性写一个大规则，风险很高：

```text
规则会变得太宽
clean_data 容易被污染
旧样本容易回归
出了问题很难定位
```

所以每一轮只解决一个清楚的问题：

```text
先诊断
再小范围修复
再 QA
再决定下一步
```

这也是为什么会有 R1、R2、R3、QA 这些任务。不是为了显得复杂，而是为了别把系统写成一锅粥。虽然人类总能把简单事写复杂，这次至少复杂得有点道理。

---

## 4. 348N 这条线是怎么来的

348N 是一个新真实样本，用来测试系统是不是只会处理前面几份 Excel。

输入是：

```text
PDF:   6862e6f3995d3dbfbed310b51601fb0a.pdf
Excel: linyang_energy_pdf_extracted_testset (1).xlsx
```

这份 Excel 是林洋能源相关的 testset workbook。

它和前面的普通 workbook 不一样，里面有大量测试集结构，而不是普通财务宽表。

第一次跑出来的核心结果是：

```text
row_count_total = 483
clean_data_row_count = 37
review_queue_row_count = 446
unknown_row_count = 367
```

大白话解释：

```text
系统能跑通，但看不懂很多行。
unknown_row_count = 367 说明 intake 层对这个 Excel 的结构不认识。
clean_data 只有 37 行，说明系统还算保守，没有把大量未知数据乱塞进去。
```

这就是 348N 后续几轮任务的起点。

---

## 5. 348N-R1：先诊断，不急着改

R1 的目的不是修代码，而是先弄清楚：367 个 unknown 到底是什么。

诊断结果发现，unknown 主要来自这些 sheet：

```text
normalized_testset
README
data_dictionary
doc_metadata
figure_index
related_research
validation_checks
market_base_data
```

最关键的是：

```text
normalized_testset 是一种长表 testset schema，不是普通财务宽表。
```

它大概长这样：

```text
record_id
source_pdf
source_page
table_name
statement
line_item
period
value
unit
value_text_original
confidence
note
```

所以不能把它当成普通财务表行。

R1 的设计意义是：

```text
先承认系统不知道，再弄清楚不知道的东西是什么。
```

这比直接写规则更安全。否则很容易把 testset 记录误判成正式财务数据，最后 clean_data 会被污染。

---

## 6. 348N-R2：识别 normalized_testset，但不让它进 clean_data

R2 的目标是解决最大的一坨 unknown：`normalized_testset`。

R2 做了什么：

```text
新增 normalized_testset header-family 检测
把 normalized_testset 行路由为 NORMALIZED_TESTSET_RECORD_ROW
保持它们 review-only
明确不让它们进入 clean_data
```

R2 前后数据：

```text
unknown_row_count: 367 -> 48
normalized_testset_record_row_count: 0 -> 320
clean_data_row_count: 37 -> 37
review_queue_row_count: 446 -> 447
pytest: 42 passed
```

大白话解释：

```text
系统终于知道 normalized_testset 是什么了。
unknown 大幅下降，但 clean_data 没增加。
这说明系统只是提高了识别能力，没有放宽 clean_data 边界。
```

为什么这样设计：

```text
normalized_testset 是测试集记录，不应该直接当成正式财务 clean data。
先把它显式分类，再放到 review-only，是最安全的做法。
```

R2 的好结果不是 unknown 下降，而是：

```text
unknown 下降的同时，clean_data 没被污染。
```

---

## 7. 348N-R2-QA：确认 R2 没越界

R2-QA 是专门审 R2 是否安全。

QA 重点看这些问题：

```text
NORMALIZED_TESTSET_RECORD_ROW 是否 review-only
normalized_testset 是否没有进入 clean_data
runner 是否只是新增统计字段
旧 wide workbook 分类是否没有回退
外部调用是否仍为 0
```

QA 结论：

```text
348N_R2_QA_CONFIRMED_NORMALIZED_TESTSET_SCHEMA_SUPPORT_VALID
```

核心结果：

```text
clean_data_row_count = 37，未变化
normalized_testset_record_row_count = 320
pytest = 42 passed
LLM / MinerU / OCR calls = 0
```

大白话解释：

```text
R2 确实只是让系统认识 normalized_testset，没有偷偷把它洗进 clean_data。
```

## 8. 348N-R3：处理剩下的非 normalized unknown

R2 之后，unknown 还剩 48 个。

这些不再是 normalized_testset，而是一些辅助类 sheet：

```text
README
data_dictionary
doc_metadata
figure_index
related_research
validation_checks
market_base_data
```

R3 的目标是把这些也明确分类。

R3 做了什么：

```text
README -> TESTSET_SUPPORTING_ROW
data_dictionary -> TESTSET_SUPPORTING_ROW
doc_metadata -> TESTSET_SUPPORTING_ROW
figure_index -> TESTSET_SUPPORTING_ROW
related_research -> TESTSET_SUPPORTING_ROW
validation_checks -> TESTSET_SUPPORTING_ROW
market_base_data -> 窄范围 MARKET_REFERENCE_ROW
normalized_testset 保持 NORMALIZED_TESTSET_RECORD_ROW 不变
```

R3 前后数据：

```text
row_count_total: 484 -> 488
unknown_row_count: 48 -> 0
clean_data_row_count: 37 -> 33
review_queue_row_count: 447 -> 455
normalized_testset_record_row_count: 320 -> 320
testset_supporting_row_count: 0 -> 49
market_reference_row_count: 2 -> 10
pytest: 48 passed
```

大白话解释：

```text
剩余 unknown 被识别清楚了。
clean_data 反而从 37 降到 33，因为之前有 4 行 README / validation_checks 不应该进 clean_data，现在被移出去了。
review_queue 增加到 455，是因为更多行被明确放到 review，而不是被强行洗白。
```

为什么这样设计：

```text
README、字段字典、图表索引、校验表，本来就不是正式财务数据。
它们可以帮助理解 testset，但不能进 clean_data。
```

market_base_data 为什么特殊：

```text
它有明确的 metric / value / unit / source-page lineage。
所以可以窄范围识别为 MARKET_REFERENCE_ROW。
但它仍然不进入 clean_data，因为当前 policy 让这些强证据市场参考行保持 REVIEW_REQUIRED。
```

这一步最重要的不是 unknown 清零，而是：

```text
unknown 清零的同时，clean_data 更保守了。
```

这说明系统不是在隐藏问题，而是在更准确地分类问题。

```text
unknown_row_count = 0
clean_data_row_count = 33
review_queue_row_count = 455

market_reference_row_count = 10
normalized_testset_record_row_count = 320

```

大白话解释：

```text

它没有把 unknown 藏起来，也没有把 supporting rows 塞进 clean_data。
```

## 10. 文档清洗：为什么要做

做到 R3-QA 后，项目文档已经很多了。

这不是完全坏事，因为 AI 协作确实需要明确的任务书和交接文档。

但问题是：

```text
AGENTS.md
CURRENT_MODEL_HANDOFF.md
.skills/
docs/codex_tasks/
docs/agent/项目进程.md
```

这些文档之间开始有重复说明。

所以做了一轮 docs cleanup。

目标不是删历史证据，而是明确每类文档的职责：

```text
AGENTS.md = 长期规则和永久边界
.skills/ = 可复用工作流规则
CURRENT_MODEL_HANDOFF.md = 当前阶段和下一步入口
docs/codex_tasks/ = 给 Codex 的完整任务书
docs/agent/ = RESULT / QA / 历史证据
docs/agent/项目进程.md = 紧凑里程碑账本
```

大白话解释：

```text
完整任务写进 docs/codex_tasks。
聊天里只给简易提示词。
新模型先读 handoff 和当前任务文档，不要全仓库乱读。
```

这一步是为了降低模型接手成本，也避免每次都复制一大堆提示词。

## 13. 下一步为什么是 R4

下一步推荐：

```text
348N-R4 clean data candidate policy review
```

R4 不是继续清 unknown。

unknown 已经是 0。

R4 要审的是：

```text
现在 clean_data 里的 33 行 qualitative_facts 是否真的应该留在那里。
```

如果这些 qualitative_facts 本质上也是 testset-specific 或 narrative-like 数据，那么它们可能不应该继续待在 clean_data。

R4 的核心问题是：

```text
clean_data 的边界是否还需要继续收紧？
```

不是为了让数据变多，而是为了确认 clean_data 真的干净。

## 15. 老 datefac/ 为什么现在不删

旧 `datefac/` 暂时不删。

原因：

```text
它是 legacy reference
里面可能还有旧 benchmark、旧 audit、旧 recovery 思路
当前 datefac_agent/ 还没完全稳定
贸然删除会制造大变更，但短期收益不大
```

现在正确做法是：

```text
datefac/ = legacy，只读参考，不默认修改
datefac_agent/ = 当前开发主线
```

等以后满足这些条件，再考虑归档或删除：

```text
active code 不再 import datefac/
测试完全不依赖 datefac/
文档里只剩历史引用
已经打 tag 或保留 archive branch
```

现在删它，风险大于收益。软件工程里最可怕的一句话就是“我觉得这个没用了”，这句话通常会在三天后变成事故报告标题。

```text
这个项目已经从“能读 Excel”进化到“能识别多种 Excel 结构，并保守地区分 clean_data 和 review_queue”。
```

```text
普通 workbook
```


这是正确的。

```text
审计管线越来越稳定

```

## 17. 一句话总结

这一轮 348N 的核心价值是：

```text
系统不再把看不懂的 testset workbook 直接当 unknown 垃圾，也没有把它们乱塞进 clean_data，而是逐步识别结构、保守分流、保留审计证据。
```

最重要的成果不是 unknown 清零。

最重要的成果是：

```text
unknown 清零的同时，clean_data 更保守，review_queue 更可解释。
```

这才是金融数据审计系统该有的样子。
