# 项目进展大白话说明

这份文档是给人看的，不是给机器看的。

它用大白话说明：DateFac Agent 现在到底在做什么、最近 R7P 到 R7X-QA 这一串任务解决了什么、还不能吹什么牛、下一步为什么要做 source-value agreement checker。

---

## 1. 这个项目现在在做什么

当前主线不是“重新抽 PDF”，也不是“训练模型”。

当前主线是：

```text
把已有 Excel 抽取结果读进来，判断哪些行可信，哪些行必须人工复核，最后形成可审计的 clean_data 和 review_queue。
```

核心流程仍然是：

```text
intake -> audit -> review -> delivery
```

大白话解释：

```text
intake = 读取 Excel，识别每一行是什么
audit = 检查证据、单位、期间、估值、row type 等问题
review = 决定进入 clean_data 还是 review_queue
delivery = 以后交付，目前 gate 还关着
```

最重要原则没有变：

```text
宁可让数据进 review_queue，也不要把不确定的数据错误放进 clean_data。
```

金融数据里，漏报可以补，错报会害人。这个项目不是为了让 clean_data 数字变好看，而是为了让结果能解释、能审计、能回滚。

---

## 2. clean_data 和 review_queue 是什么

`clean_data` 是系统认为相对可信、可以作为后续结构化数据使用的行。

`review_queue` 是需要人工复核的行。

大白话说：

```text
clean_data = 暂时能放行的行
review_queue = 不能自动放行、需要人看一眼的行
```

review_queue 大不一定坏。只要它可解释，就比把脏数据塞进 clean_data 好。

---

## 3. 当前验证过哪些 workbook family

目前已经验证过三类真实 workbook family：

```text
Anjing / 安井
Linyang / 林洋
Taihao / 泰豪
```

### 安井

```text
R7U clean_data_row_count = 65
R7U review_queue_row_count = 17
market_reference_row_count = 10
R7S removed normal fact rows = 0
```

大白话解释：正常财务数据还在，market reference 行不再混进 clean_data，R7S 没误伤安井。

### 林洋

```text
R7U clean_data_row_count = 0
R7U review_queue_row_count = 489
unknown_row_count = 0
market_reference_row_count = 10
```

大白话解释：林洋是 testset / supporting / market reference 很重的 workbook。它的安全结果是 clean_data 保持 0，review_queue 很大但可解释。

### 泰豪

```text
R7Q clean_data_row_count = 92
R7Q review_queue_row_count = 66
R7T clean_data_row_count = 72
R7T review_queue_row_count = 86
unknown_row_count = 0
market_reference_row_count = 2
```

大白话解释：R7S 后，20 个不该自动放行的 strict-table scaffolding / pseudo-header / comparison 行从 clean_data 移入 review_queue。

典型风险行包括：

```text
市场数据
厂商
对比维度
订单日期
项目
指标
```

这些行更像表头、对比维度或表格骨架，不是稳定的财务 fact。它们不该混在 clean_data 里装成熟人，金融数据不是化妆舞会。

## 4. R7P 到 R7V：clean-boundary 收束

### R7P / R7P-FIX / R7P-FIX2：修 market reference 外泄

最初 Taihao pilot 暴露出一个问题：

```text
MARKET_REFERENCE_ROW 里的“收盘价”等行进入了 clean_data
```

R7P-FIX 找到 root cause：

```text
MARKET_REFERENCE_ROW + WEAK_EVIDENCE 被 clean_candidate_policy 放进 INTERNAL_REFERENCE_CANDIDATE
```

R7P-FIX2 修正为：

```text
MARKET_REFERENCE_ROW -> REVIEW_REQUIRED
```

大白话解释：市场参考数据有价值，但不能自动当成 clean financial fact 放行。

### R7Q / R7R / R7S：修 strict-table 伪表头和对比行过宽

Market reference 修完后，R7Q 又发现一个新风险：

```text
STRICT_FINANCIAL_TABLE_ROW 里，有些 pseudo-header / comparison-dimension / scaffolding 行仍可能进入 clean_data
```

R7R 的设计结论是：

```text
不新增 row_type
不改 row_type_classifier
在 clean_candidate_policy 层收窄 clean admission
```

R7S 实现后，规则变得更保守：

```text
STRICT_FINANCIAL_TABLE_ROW + WEAK_EVIDENCE + 全非数字 period_values / period label 回声
-> REVIEW_REQUIRED
```

### R7T / R7U / R7V：证明修复不是只对一个样本有效

R7T 重跑 Taihao：

```text
clean_data_row_count: 92 -> 72
review_queue_row_count: 66 -> 86
20 个风险行移出 clean_data
无新 guardrail failure
```

R7U 回归 Linyang / Anjing：

```text
Linyang: clean_data = 0，review_queue = 489，无回归
Anjing: clean_data = 65，review_queue = 17，R7S 移除 0 个正常 fact 行
```

R7V 总结：

```text
Taihao / Linyang / Anjing 跨 family clean-boundary 验证通过
但 readiness gates 继续关闭
```

大白话解释：系统已经证明“收窄 clean_data 边界”不是拍脑袋，也不是只修一个样本。但这仍然不等于生产可用。

## 5. 为什么现在还不是 production ready

当前 gates 仍然是：

```text
client_ready = false
production_ready = false
formal_client_export_allowed = false
demo_export_only = true
```

原因很现实：

```text
clean-boundary 变稳了，但 evidence strength 还不够强
```

尤其是 Taihao，很多行仍是：

```text
WEAK_EVIDENCE
```

`WEAK_EVIDENCE` 不等于错，但它表示：

```text
有 workbook lineage，但还没有足够强的、可审计的 source-value agreement
```

也就是说，系统知道这行来自哪个 Excel、哪个 sheet、哪一行、哪个 metric，但还不能严肃地说：

```text
我已经核验 PDF 指定页上的原始证据，且数值一致。
```

所以现在不能打开 production_ready。把弱证据包装成强证据，那不叫工程推进，叫给事故写邀请函。

## 6. R7W / R7X / R7X-QA：证据增强第一阶段

R7W 的核心结论是：

```text
Strong evidence must mean traceable and auditable, not merely plausible.
```

强证据不能只是：

```text
模型觉得像
有个看起来像页码的字段
Excel 里有出处文字
```

R7X 做了两件事：

```text
1. 从 explicit_evidence_ref 里解析 page_number
2. 增加 agreement_status 字段
```

支持的页码形式包括：

```text
第12页
12页
页码：12
page 12
Page 12
p.12
P12
pp. 12-13
第12-13页
```

`agreement_status` 当前语义是：

```text
MISSING = 没有 explicit evidence / provenance anchor
UNVERIFIED = 有 provenance，但还没有做 source-value agreement check
VERIFIED = 未来 deterministic checker 确认数值一致
DISAGREED = 未来 deterministic checker 确认数值不一致
```

R7X 当前只填：

```text
MISSING
UNVERIFIED
```

R7X-QA 已确认：

```text
Decision = 348N_R7X_QA_CONFIRMED_EVIDENCE_PROVENANCE_PARSING_VALID
tests/agent = 95 passed
page_number_parsing_result = valid
agreement_status_result = valid
strong_evidence_claim_result = valid
readiness_gates = closed
```

最关键边界：

```text
解析出 page_number + agreement_status = UNVERIFIED
仍然保持 WEAK_EVIDENCE
不能自动变成 STRONG_EVIDENCE
```

大白话解释：

```text
知道该去哪一页找证据，不等于已经核验证据。
```

这点非常重要。否则系统会把“知道门牌号”吹成“已经进屋验货”，离谱但很常见，毕竟软件系统最擅长一本正经地自我感动。

## 7. 当前最重要的项目状态

现在项目已经完成了两条重要收束：

```text
clean-boundary 收束：R7P -> R7V
证据 provenance 起步：R7W -> R7X-QA
```

当前最准确的描述是：

```text
系统已经能跨多个 workbook family 保守区分 clean_data / review_queue。
系统已经开始把 evidence provenance 结构化。
系统已经确认 page_number parsed 不等于 VERIFIED。
但还没有完成 source-value agreement verification。
```

所以当前还不是：

```text
生产可用系统
正式客户交付系统
完全自动化金融数据抽取系统
```

当前更像是：

```text
一个严格审计型、可复现、保守边界明确的 Excel intake/audit pipeline
```

## 8. 当前下一步：R7Y

当前下一步是：

```text
348N-R7Y deterministic source-value agreement checker / evidence agreement verification slice
```

建议思考程度：

```text
recommended_reasoning_level = max
reason = R7Y 会触及 VERIFIED / DISAGREED 的真实证据核验语义，错误实现会污染后续 evidence_strength 和 readiness 判断。
```

R7Y 要做的事：

```text
从“我知道证据可能在哪一页”推进到“我能确定结构化数值和来源文本一致”。
```

R7Y 仍然不能做的事：

```text
不能运行 OCR / LLM / VLM
不能打开 production_ready
不能让 VERIFIED 自动等于 clean admission
不能让 VERIFIED 自动等于正式客户交付
不能改变 MARKET_REFERENCE_ROW / qualitative_facts / R7S clean-boundary policy
```

如果 source text 不足、数值不完整、文本歧义、只找到部分值，都应该保守地保持：

```text
UNVERIFIED
```

而不是硬吹：

```text
VERIFIED
```

这就是下一步的重点：让系统真正开始核验证据，而不是继续拿“看起来有出处”自我感动。

## 10. 一句话总结

这一阶段的核心进展是：

```text
项目已经把 clean_data 边界从“能跑”推进到“跨 workbook family 更保守、更可解释”，并把证据链从 WEAK_EVIDENCE 推进到 page provenance + agreement_status。
```

但当前仍然必须记住：

```text
page_number parsed != VERIFIED
UNVERIFIED != STRONG_EVIDENCE
clean_data 更干净 != production ready
```

项目正确姿势仍然是：

```text
先做可审计，再谈自动化；先保守 review，再谈正式交付。
```
