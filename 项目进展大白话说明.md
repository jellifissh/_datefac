# 项目进展大白话说明

这份文档是给人看的，不是给机器看的。

它用大白话说明：DateFac Agent 现在到底在做什么、最近 R7P 到 R7Y 这一串任务解决了什么、还不能吹什么牛、下一步为什么要做 R7Y-QA。

---

## 1. 这个项目现在在做什么

当前主线不是重新抽 PDF，也不是训练模型。

当前主线是：

```text
把已有 Excel 抽取结果读进来，判断哪些行可信，哪些行必须人工复核，最后形成可审计的 clean_data 和 review_queue。
```

核心流程仍然是：

```text
intake -> audit -> review -> delivery
```

最重要原则没有变：

```text
宁可让数据进 review_queue，也不要把不确定的数据错误放进 clean_data。
```

金融数据里，漏报可以补，错报会害人。这个项目不是为了让 clean_data 数字变好看，而是为了让结果能解释、能审计、能回滚。

---

## 2. clean_data 和 review_queue 是什么

```text
clean_data = 暂时能放行的行
review_queue = 不能自动放行、需要人看一眼的行
```

review_queue 大不一定坏。只要它可解释，就比把脏数据塞进 clean_data 好。

## 4. R7P 到 R7V：clean-boundary 收束

R7P 最初暴露的问题：

```text
MARKET_REFERENCE_ROW 里的“收盘价”等行进入了 clean_data
```

R7P-FIX2 修正为：

```text
MARKET_REFERENCE_ROW -> REVIEW_REQUIRED
```

Market reference 修完后，R7Q 又发现：

```text
STRICT_FINANCIAL_TABLE_ROW 里，有些 pseudo-header / comparison-dimension / scaffolding 行仍可能进入 clean_data
```

R7S 收窄后，Taihao 结果变成：

```text
clean_data_row_count: 92 -> 72
review_queue_row_count: 66 -> 86
20 个风险行移出 clean_data
```

典型风险行包括：

```text
市场数据
厂商
对比维度
订单日期
项目
指标
```

这些行更像表头、对比维度或表格骨架，不是稳定的财务 fact。它们不该混在 clean_data 里装成熟人，金融数据不是化妆舞会。

R7V 总结：

```text
Taihao / Linyang / Anjing 跨 family clean-boundary 验证通过
但 readiness gates 继续关闭
```

## 5. 为什么现在还不是 production ready

当前 gates 仍然是：

```text
client_ready = false
production_ready = false
formal_client_export_allowed = false
demo_export_only = true
```

原因很现实：

```text
clean-boundary 变稳了，但 evidence strength 还要继续验证
```

`WEAK_EVIDENCE` 不等于错，但它表示：

```text
有 workbook lineage，但还没有足够强的、可审计的 source-value agreement
```

所以不能把弱证据包装成强证据。那不叫工程推进，叫给事故写邀请函。

---

## 6. R7W / R7X / R7X-QA：证据 provenance 起步

R7W 的核心结论是：

```text
Strong evidence must mean traceable and auditable, not merely plausible.
```

R7X 做了两件事：

```text
1. 从 explicit_evidence_ref 里解析 page_number
2. 增加 agreement_status 字段
```

`agreement_status` 语义：

```text
MISSING = 没有 explicit evidence / provenance anchor
UNVERIFIED = 有 provenance，但还没有做 source-value agreement check
VERIFIED = deterministic checker 确认数值一致
DISAGREED = deterministic checker 确认数值不一致
```

R7X-QA 已确认：

```text
page_number parsed != VERIFIED
UNVERIFIED != STRONG_EVIDENCE
readiness_gates = closed
tests/agent = 95 passed
```

大白话解释：

```text
知道该去哪一页找证据，不等于已经核验证据。
```

这点非常重要。否则系统会把“知道门牌号”吹成“已经进屋验货”，离谱但很常见。

---

## 7. R7Y：source-value agreement checker 已实现

R7Y 做的是下一步：

```text
从“我知道证据可能在哪一页”推进到“我能确定结构化数值和来源文本一致”。
```

R7Y 实现了 deterministic source-value agreement checker。

结果：

```text
commit = 973ff68 feat: add deterministic evidence agreement checker
files_modified = 2
pytest tests/agent -q = 106 passed
```

R7Y 支持的 numeric normalization：

```text
1234
1,234
1234.56
1,234.56
-123
(123)
12.3%
12.30%
```

R7Y 的保守语义：

```text
no explicit/page provenance -> MISSING
explicit/page provenance but no source_text -> UNVERIFIED
all row numeric period_values match source_text -> VERIFIED
source_text has numbers but none match -> DISAGREED
partial coverage -> UNVERIFIED
text-only facts -> UNVERIFIED
```

最重要边界：

```text
VERIFIED != STRONG_EVIDENCE
VERIFIED != clean admission
VERIFIED != production_ready
DISAGREED 也不是拍脑袋 mismatch，而是确定性 numeric comparison 证明不一致
```

这一步让系统开始真正做“数值核验”，不是继续拿“看起来有出处”自我感动。

---

## 8. 当前最重要的项目状态

现在项目已经完成了两条重要收束：

```text
clean-boundary 收束：R7P -> R7V
证据增强起步：R7W -> R7Y
```

当前最准确的描述是：

```text
系统已经能跨多个 workbook family 保守区分 clean_data / review_queue。
系统已经开始把 evidence provenance 和 numeric agreement 结构化。
但 R7Y 还需要 QA，不能直接拿实现结果当最终可信结论。
```

所以当前还不是：

```text
生产可用系统
正式客户交付系统
完全自动化金融数据抽取系统
```

当前更像是：

```text
一个严格审计型、可复现、保守边界明确、正在补强证据链的 Excel intake/audit pipeline
```

## 9. 当前下一步：R7Y-QA

当前下一步是：

```text
348N-R7Y-QA deterministic source-value agreement checker review
```

建议思考程度：

```text
recommended_reasoning_level = max
reason = R7Y-QA 审查 VERIFIED / DISAGREED 的证据一致性语义，错误放行会污染后续 evidence_strength 和 readiness 判断。
```

R7Y-QA 要审：

```text
VERIFIED 是否只在所有 numeric period_values 被 source_text 覆盖时产生
DISAGREED 是否只在有 source numeric tokens 且完全无匹配时产生
partial coverage 是否保持 UNVERIFIED
text-only facts 是否保持 UNVERIFIED
VERIFIED 是否没有自动变成 STRONG_EVIDENCE
VERIFIED 是否没有自动进入 clean_data
VERIFIED 是否没有打开 readiness gates
MARKET_REFERENCE_ROW / qualitative_facts / R7S clean-boundary 是否未回归
```

如果 R7Y-QA 通过，下一步才考虑：

```text
R7Z：把 source_text 输入接入真实 pipeline / evidence index，或先做 targeted fixture coverage
```

不要从 R7Y 直接跳到 production readiness。人类最爱把“单元测试过了”翻译成“可以上线了”，这通常是事故文学的第一章。

---

## 10. 一句话总结

这一阶段的核心进展是：

```text
项目已经把 clean_data 边界从“能跑”推进到“跨 workbook family 更保守、更可解释”，并把证据链从 page provenance 推进到 deterministic numeric source-value agreement checker。
```

但当前仍然必须记住：

```text
page_number parsed != VERIFIED
UNVERIFIED != STRONG_EVIDENCE
VERIFIED != production_ready
clean_data 更干净 != 正式交付
```

项目正确姿势仍然是：

```text
先做可审计，再谈自动化；先保守 review，再谈正式交付。
```
