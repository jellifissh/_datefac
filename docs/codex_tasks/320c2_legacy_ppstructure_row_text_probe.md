## task_title
Integrate legacy PPStructure output as row-text recognition probe and test row-text-to-metric-candidate extraction

## project

Completed stages:

- 320B: MinerU benchmark runner.
- 320B2: MinerU benchmark diagnostics/calibration.
- 320C: TableAsset recognition probe shell.

Latest 320B2 benchmark:

320C probe selected table assets successfully, but recognizer was `none`.

Manual local recognizer experiments:
- PaddleOCR 3.6 `TableStructureRecognition` failed on Windows CPU/Paddle backend.
- PaddleOCR 3.6 `TableRecognitionPipelineV2` also failed on oneDNN/Paddle backend even after English cache path and mkldnn flags.
- Legacy PaddleOCR `PPStructure` in a separate `ppstructure_legacy` environment finally produced outputs from MinerU table JPGs.
- The output contains `res_0.txt` and an `.xlsx` file.
- However, the legacy PPStructure result usually does not preserve full column grid. It often outputs one row as one text-heavy cell.

Engineering conclusion:
Legacy PPStructure is useful as a row-text recognizer, not yet a reliable full grid recognizer. The next task should formalize this route:

MinerU TableAsset image -> legacy PPStructure result -> ExtractedTable with row_text -> row-text parsing probe -> MetricCandidate preview.

This is still sandbox-only. Do not update production delivery files.

## goal
Implement a sandbox-only 320C2 probe that can ingest legacy PPStructure result files and convert them into DateFac-friendly row-text extracted tables, then attempt simple financial row parsing.

The task has two layers:
1. Parse legacy PPStructure output into `ExtractedTable` records.
2. Try deterministic row-text-to-candidate extraction for core financial metrics.

This task must not require a perfect cell grid. Row text is acceptable and expected.

## legacy_ppstructure_result_reader requirements
Implement a defensive reader for PPStructure output.

It should support:
- `res_*.txt` that contains Python/JSON-like structure from PPStructure;
- generated xlsx files where row text may appear mostly in the first column;
- optional html if present.

For `res_*.txt`:
- try `json.loads` if JSON;
- if not valid JSON, try `ast.literal_eval` defensively;
- if both fail, parse as plain text and record a warning.

For xlsx:
- read all sheets with pandas/openpyxl;
- preserve row order;
- collapse non-empty cells into row_text;
- keep raw row/cell values.

Do not crash on malformed files. Record warnings.

## row_text_metric_extractor requirements
Create deterministic extraction from row text.

Target first-phase metrics:
- EPS / 每股收益
- ROE / 净资产收益率
- PE / P/E / 市盈率
- PB / P/B / 市净率
- EV/EBITDA
- Revenue / 营业收入
- Net Profit / 净利润 / 归属于母公司净利润 / 归母净利润
- Gross Margin / 毛利率
- Revenue Growth / 收入增长
- Net Profit Growth / 净利润增长率
- Debt Ratio / 资产负债率
- Operating Cash Flow / 经营活动现金流

Target years:
- 2024
- 2025
- 2026E
- 2027E
- 2028E
- also accept 2024A/2025A where present.

Parsing strategy:
1. Detect header year sequence from row_texts if available.
2. If header not found, use canonical sequence: 2024, 2025, 2026E, 2027E, 2028E, but mark `YEAR_INFERRED` warning.
3. For each row, detect metric label by keyword/synonym.
4. Extract numeric tokens, including:
   - decimals: `3.39`
   - percentages: `15%`, `15.2%`
   - comma numbers: `12,488`
   - parenthesized negatives: `(967)` -> `-967`
   - Chinese full-width parentheses if present.
5. Align numeric token count to detected years.
6. Generate candidate rows with risk tags.

Do not over-trust output. This is a probe.
