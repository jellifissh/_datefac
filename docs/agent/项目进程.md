## 5. 348A-R4：Clean Data Candidate Policy

核心原则：

```text
internal clean-data candidate ≠ client_ready
internal clean-data candidate ≠ production_ready
internal clean-data candidate ≠ formal client delivery
```

主要规则：

```text
STRICT_FINANCIAL_TABLE_ROW + WEAK_EVIDENCE + 无 unit/period/valuation issue
=> INTERNAL_CLEAN_CANDIDATE

MARKET_REFERENCE_ROW + WEAK_EVIDENCE + 无 unit issue
=> INTERNAL_REFERENCE_CANDIDATE

NARRATIVE_ASSERTION
=> NARRATIVE_REVIEW，不进 clean_data
```

结果：

```text
clean_data_row_count = 75
review_queue_row_count = 7
internal_clean_candidate_count = 65
internal_reference_candidate_count = 10
narrative_review_count = 5
review_required_count = 2
client_ready = false
production_ready = false
formal_client_export_allowed = false
```

结论：第一真实样本产生内部可用 clean-data candidate，但没有打开正式交付 gate。

## 8. 348S-R1：Intake Schema Generalization

第二 workbook 结构特点：

```text
报告概要 是两列 key-value summary sheet
盈利预测与估值 / 资产负债表 / 利润表 / 现金流量表 的真实 header row 在第 3 行
重要财务与估值指标 是 strict financial-style sheet
可比公司估值 是 market/reference sheet
盈利预测分业务 是 structured forecast table
```

修改方向：

```text
新增 header-row detection
新增 key-value summary sheet 支持
扩展 strict / market / narrative sheet mapping
增强 period label search
```

第二样本改善结果：

```text
row_count_total: 119 -> 112
unknown_row_count: 53 -> 0
period_issue_count: 66 -> 5
clean_data_row_count: 0 -> 87
review_queue_row_count: 119 -> 25
internal_clean_candidate_count = 75
internal_reference_candidate_count = 12
narrative_review_count = 18
review_required_count = 5
excluded_from_clean_data_count = 2
```

baseline 回归结果：

```text
row_count_total = 82
unit_issue_count = 0
period_issue_count = 2
unknown_row_count = 0
clean_data_row_count = 75
review_queue_row_count = 7
```

结论：R1 显著改善第二 workbook 泛化，同时第一样本 baseline 未明显回退。

## 10. 348S-R2：Unit/Period Residual Refinement

```text

```
